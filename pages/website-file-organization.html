Why We Organize Our Website Files

In class, we discussed why we keep our HTML and CSS in separate files. This page expands on that concept and explains the professional file structure we use for our portfolio websites.

The "Why": A Tidy House vs. A Messy Room

Imagine your website is a house.

  • HTML is the blueprint and the frame: It defines the rooms and their purpose (a <header> is the entryway, a <footer> is the foundation, a <p> is a block of text).
  • CSS is the interior design: It's the paint colors, the furniture, the lighting, and all the decorations that make the house look good.
  • JavaScript is the utilities: It's the electricity and plumbing that make things interactive, like light switches or a working doorbell.

You could technically pile all your furniture and paint cans inside the frame of the house as you build it (this is like putting CSS inside your HTML file). But a professional builder would never do that. They keep the blueprints, the building materials, and the decorations separate until they are needed. This is what we do when we build websites.

This principle is called "Separation of Concerns," and it has four major benefits:

  • Reusability: As mentioned in class, you can use one CSS file (styles.css) to define the look and feel of your entire website. Every page can link to the same file to get the same fonts, colors, and layout, saving you from writing the same code over and over.
  • Maintainability: If you want to change the main color of your website, you only have to change it in one place (styles.css) instead of hunting it down in every single HTML file. This makes updates and fixing mistakes much easier.
  • Readability: When your HTML file only contains content structure and your CSS file only contains styling rules, the code is much cleaner and easier for you, your collaborators, and even AI to read and understand.
  • Collaboration: In larger projects, a designer might work on the CSS files while a content manager works on the HTML files. Keeping them separate allows people to work on the same website at the same time without conflict.

Anatomy of Your Project Folder

Your portfolio website uses a standard, professional folder structure. Think of the main project-root/ as your house, and everything inside is a specific, well-organized room or filing cabinet.

project-root/
├─ css/
│  ├─ styles.css
│  └─ (page-specific css files)
├─ scripts/main.js
├─ images/
├─ icons/
├─ collections/
│  ├─ index.html
│  └─ <collection-name>/
│     ├─ index.html
│     └─ <work-name>/
│        ├─ index.html
│        ├─ models/
│        └─ images/
├─ 404.html
├─ index.html
├─ .gitignore
├─ robots.txt
├─ site-structure.md
└─ prompt-for-jules.md

Key Folders

/css/:
This is where all your styling rules live.
  • styles.css is your global stylesheet. It controls the look of elements that appear on every page, like your navigation menu, footer, fonts, and color scheme.
  • Page-specific files (e.g., contact.css) are for styles that only apply to one page, keeping your global file clean.
/scripts/:
This folder holds your JavaScript files. main.js will contain the code for interactive elements like a mobile menu or animations.
/images/:
This is for site-wide assets only, like your logo or a background texture. Artwork images do not go here.
/icons/:
A special place for your favicon files (the small icon that appears in the browser tab).
/collections/:
This is your main art gallery. It's carefully structured so that each collection and each individual work has its own "room," complete with its own page (index.html) and folders for its specific images and 3D models.

Key Files (in the root)

index.html:
This is the front door of your website. When a browser visits your domain (e.g., yourname.com), this is the first file it looks for.
404.html:
This is the "Page Not Found" page. If a visitor tries to go to a page that doesn't exist, they will see this instead of an ugly browser error.
.gitignore:
This is a special file that tells Git (and GitHub) which files or folders to ignore. This is useful for keeping sensitive information or temporary files out of your public repository.
robots.txt:
This file gives instructions to search engine bots (like Googlebot) about which pages on your site they should or should not crawl and index.
site-structure.md & prompt-for-jules.md:
These are your project's architectural blueprints. They document the planned structure and the exact instructions given to the AI, which is incredibly useful for reference.

File and Folder Naming Conventions

How you name your files is just as important as where you put them. Following these rules will prevent errors and keep your project manageable.

  • Use lowercase file names with no spaces. This is the most important rule. Web servers can be case-sensitive, and spaces in filenames can cause links to break. If you need to separate words, use a dash.
    • Do: contact-us.html, main-logo.png
    • Don't: Contact Us.html, Main Logo.png
  • Always name your homepage index.html. When a browser visits a folder, index.html is the default file it looks for automatically. Every folder that acts as a "page" in your site (like the root folder, a collection folder, or a work folder) should have its own index.html.
  • Be consistent and descriptive. Give your files logical names so you know what they are at a glance.
    • Good: cv-spring-2025.pdf, gallery-portrait-01.jpg
    • Bad: doc1.pdf, image123.jpg
  • Keep file extensions standard. Use the correct, common extension for each file type so browsers and operating systems know how to handle them (.html, .css, .js, .jpg, .png, etc.).

Back up your files regularly. Using GitHub is the best way to do this, as it saves a complete history of your project.

How Files Find Each Other: Relative Paths

When you link your CSS file in your HTML, you use a path. A path is simply the direction from one file to another.

  • Absolute Path: An absolute path starts from the very root of a domain (e.g., /css/styles.css). This is like giving someone your full street address. It works, but only if your website is at the very top level of a domain.
  • Relative Path: A relative path starts from the current file's location (e.g., css/styles.css). This is like saying, "Go out of this room and find the 'css' room down the hall."

Why We MUST Use Relative Paths:

For this class, we use GitHub Pages to host our websites. GitHub often places your site in a subdirectory (e.g., yourname.github.io/my-portfolio/). If you used an absolute path like /css/styles.css, the browser would look for it at yourname.github.io/css/styles.css and wouldn't find it, breaking your site's styling.

By using a relative path like css/styles.css, the browser correctly looks for it inside your project folder, no matter where it's hosted. This makes your site portable and ensures all your links, images, and styles work correctly.

Examples of Relative Paths

Let's look at a few common scenarios from our project structure.

Scenario 1: Linking the global stylesheet from your homepage.

Current File: index.html (in the root)

Target File: css/styles.css

The Path: href="css/styles.css"

Why: You are in the root folder. The browser just needs to look inside the css folder that is right there and find the styles.css file.

Scenario 2: Linking an artwork image from an individual work page.

Current File: /collections/residue/vessel-01/index.html

Target File: /collections/residue/vessel-01/images/front-view.jpg

The Path: src="images/front-view.jpg"

Why: You are already inside the vessel-01 folder. You just need to tell the browser to look inside the images folder (which is right next to your index.html file) and grab the front-view.jpg file.

Scenario 3: Linking the global stylesheet from a deeply nested work page.

Current File: /collections/residue/vessel-01/index.html

Target File: /css/styles.css (in the root)

The Path: href="../../../css/styles.css"

Why: This path looks tricky, but it's just giving step-by-step directions. Each ../ tells the browser to "go up one level."

  • From vessel-01/, the first ../ goes up to the residue/ folder.
  • The second ../ goes up to the collections/ folder.
  • The third ../ goes up to the project-root/ folder.

Now that it's in the root, the browser can look for the css/ folder and find styles.css.
